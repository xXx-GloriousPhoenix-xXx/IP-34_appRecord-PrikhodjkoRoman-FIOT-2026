## 2. Аналіз предметної області

### 2.1 Актуальність теми
В умовах зростання популярності віддаленої роботи та фрілансу, коворкінги стають невід'ємною частиною інфраструктури сучасних міст. Щодня тисячі людей шукають місця для комфортної роботи з високошвидкісним інтернетом.

Існує проблема неефективного управління робочими місцями: адміністратори витрачають час на телефонні дзвінки, бронювання губляться, виникають ситуації "double-booking" (подвійного бронювання), а клієнти не мають прозорого механізму самостійного обрання місця.

Розв'язання цієї проблеми є важливим, оскільки автоматизація процесу бронювання підвищує задоволеність клієнтів, збільшує завантаженість коворкінгу та зменшує навантаження на персонал.

### 2.2 Об’єкт і предмет
**Об’єкт:** процес оренди та бронювання робочих місць у просторі колективної роботи (коворкінгу).

**Предмет:** програмні засоби та алгоритми для автоматизації бізнес-процесів коворкінгу, а також моделі даних для зберігання інформації про користувачів, робочі місця та фінансові операції.

### 2.3 Практичне значення
Результатом роботи є прототип вебзастосунку для бронювання робочих місць у коворкінгу з адаптивним інтерфейсом та сформованою моделлю предметної області. Отриманий результат може використовуватися як основа для подальшої реалізації backend-частини (API, база даних, авторизація, ролі) та розширення функціональності (панель адміністратора, аналітика, нотифікації).

### 2.4 Завдання роботи
1. Обрати предметну область (система бронювання коворкінгу) та описати проблематику.
2. Визначити акторів системи та межі взаємодії.
3. Сформувати бізнес-логіку роботи системи.
4. Сформувати перелік функціональних вимог (FR).
5. Сформувати перелік нефункціональних вимог (NFR).
6. Побудувати Use-case діаграму та описати сценарії використання.
7. Побудувати ER-діаграму даних (логічну модель).
8. Побудувати компонентну діаграму.
9. Реалізувати адаптивний інтерфейс (header/main/footer, адаптивне меню, адаптивна сітка контенту).
10. Підготувати скріншоти та фрагменти коду для звіту.

## 3. Моделювання системи

### 3.1 Актори системи
| Актор | Опис |
|-------|------|
| **Unauthorized (Неавторизований)** | Має можливість перегляду доступних місць для бронювання коворкінгу, перегляду доступних тарифів та створення акаунту. |
| **User (Користувач)** | Має можливість бронювання місць, скасування бронювань, оформлення абонементу, перегляду історії своїх бронювань та оплати рахунків. |
| **Moderator (Модератор)** | Має можливість додавати/видаляти/редагувати робочі місця для бронювання, змінювати їх статус (активне/неактивне), переглядати всі бронювання системи. |
| **System (Система)** | Автоматичний актор, який виконує фонові процеси: перевірку прострочених бронювань (скасування через 48 годин без оплати), деактивацію абонементів після закінчення терміну дії. |

### 3.2 Бізнес-логіка
1. **Правило бронювання:** Користувач може створити бронювання на конкретний проміжок часу. Система перевіряє, чи не перетинається бажаний час з існуючим активним бронюванням для обраного робочого місця. Забороняється створювати конфліктуючі бронювання.

2. **Правило знижки:** Якщо загальна тривалість бронювання перевищує 4 години, до загальної вартості автоматично застосовується знижка у розмірі 10%.

3. **Правило скасування:** Користувач може скасувати бронювання. Якщо до моменту початку бронювання залишилось менше 2 годин, скасування стає неможливим.

4. **Правило абонемента:** Користувач може придбати місячний, квартальний або річний абонемент лише за умови відсутності активних боргів за попередніми бронюваннями та за умови відсутності активного абонемента.

5. **Правило терміну дії:** Абонемент є активним встановлений термін (30, 90 або 365 днів) з моменту активації, після чого автоматично деактивується.

6. **Правило прострочення:** Бронювання, за яким не було здійснено оплату протягом 48 годин з моменту створення, автоматично скасовується системою.

|----|--------|------|-------|

| FR-02 | Перегляд робочих місць | Відображення списку доступних робочих місць з їх характеристиками (тип, місткість, ціна). | Unauthorized, User, Moderator |
| FR-03 | Створення бронювання | Вибір місця, дати та часу з автоматичним розрахунком вартості з урахуванням знижки та перевіркою наявності. | User |
| FR-04 | Скасування бронювання | Зміна статусу бронювання на "cancelled" з перевіркою часових обмежень (не менше 2 годин до початку). | User |
| FR-05 | Оплата бронювання | Позначення бронювання як оплаченого після проведення платежу. | User |
| FR-06 | Перегляд історії бронювань | Відображення всіх бронювань користувача з їх статусами та деталями. | User |
| FR-07 | Оформлення абонемента | Покупка абонемента (місячний/квартальний/річний) з перевіркою відсутності боргів та активного абонемента. | User |
| FR-08 | Управління робочими місцями | Додавання, редагування, видалення та зміна статусу робочих місць. | Moderator |
| FR-09 | Автоматичне скасування | Фоновий процес скасування неоплачених бронювань через 48 годин. | System |
| FR-10 | Автоматична деактивація | Фоновий процес деактивації абонементів після закінчення терміну дії. | System |

### 3.7 ER-модель даних
![ER](/assets/labs/lab-1/ER.png)
**Рис. 2 – ER-діаграма.**

**Пояснення сутностей:**
- **User** — користувач системи з персональними даними (ім'я, email, телефон, вік, статус боргу).
- **Workspace** — робоче місце або переговорна кімната з характеристиками (тип, місткість, ціна за годину, статус активності).
- **Booking** — бронювання, що зв'язує користувача та робоче місце на певний проміжок часу з фінансовими даними.
- **Subscription** — абонемент користувача з терміном дії, ціною та статусом.

**Зв'язки:**
- User 1 : N Booking (один користувач може мати багато бронювань)
- Workspace 1 : N Booking (одне місце може мати багато бронювань)
- User 1 : N Subscription (один користувач може мати багато абонементів)

### 3.8 Діаграма компонентів
![Components](/assets/labs/lab-1/Components.png)
**Рис. 3 – Компонентна діаграма.**

**Опис компонентів:**
- **Frontend Client** — клієнтська частина (HTML/CSS/JS), що забезпечує адаптивний інтерфейс та взаємодію з користувачем.
- **REST API** — серверна частина (Express.js), що обробляє HTTP-запити та реалізує бізнес-логіку.
- **Database (Postgres)** — реляційна база даних для зберігання інформації про користувачів, місця, бронювання та абонементи.

## 4. Реалізація адаптивного інтерфейсу

### 4.1 Загальна структура сторінок
Інтерфейс побудовано на базових структурних блоках: **Header**, **Main**, **Footer**. Навігація доступна з будь-якої сторінки, а ключові сценарії винесені на окремі маршрути:

- `/` — головна сторінка з переглядом робочих місць
- `/bookings` — сторінка управління бронюваннями
- `/subscriptions` — сторінка абонементів
- `/users` — сторінка користувачів
- `/workspaces` — сторінка управління робочими місцями

### 4.2 Фрагмент коду: адаптивне бургер-меню (Header)
На екранах `max-width: 768px` бургер-меню відображається, на більших екранах — стандартна горизонтальна навігація.

```
<button class="menu-toggle" id="menuToggle">
    <i class="fas fa-bars"></i>
</button>

<script>
const menuToggle = document.getElementById('menuToggle');
const sidebar = document.getElementById('sidebar');

menuToggle.addEventListener('click', () => {
    sidebar.classList.add('active');
});

document.getElementById('closeSidebar').addEventListener('click', () => {
    sidebar.classList.remove('active');
});
</script>

@media (max-width: 768px) {
    .sidebar {
        transform: translateX(-100%);
    }
    
    .sidebar.active {
        transform: translateX(0);
    }
    
    .menu-toggle {
        display: block;
    }
    
    .main-content {
        margin-left: 0;
    }
}
```

## 6. Висновки
У межах лабораторної роботи виконано вибір предметної області (система бронювання коворкінгу) та сформовано модель майбутньої інформаційної системи. Обґрунтовано актуальність теми, визначено мету, завдання, об’єкт і предмет, описано бізнес-логіку процесу бронювання робочих місць та оформлення абонементів. Сформовано функціональні та нефункціональні вимоги, побудовано Use-case, ER та компонентну діаграми.
Реалізовано адаптивний інтерфейс вебзастосунку з базовими структурними компонентами (header/main/footer), бургер-меню на мобільних екранах та навігацією без помилок 404. Клієнтська частина повністю інтегрована з REST API через JSON-обмін даними.
Отриманий результат є основою для подальшого розвитку системи: підключення бази даних Postgres через Sequelize, реалізація повноцінної авторизації з ролями (Unauthorized, User, Moderator), додавання системи сповіщень та адміністративної панелі.
